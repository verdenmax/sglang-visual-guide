What is SGLang

SGLang is a high-performance serving engine for large language models and multimodal models: the model is already trained, and what SGLang solves is how to run it online fast and cheaply — generating tokens at low latency and high throughput under thousands of concurrent requests. It comes in two halves: a frontend language (DSL) that lets you orchestrate LLM calls like a program, and a runtime engine that actually executes the requests. This lesson sketches its outline from 10,000 feet; later lessons zoom into the engine layer by layer.

🔌 Analogy
Picture SGLang as a busy restaurant kitchen. If you fired up a fresh pan and prepped every ingredient from scratch for each guest, the kitchen would collapse. A smart kitchen does two things: first, it merges orders that arrive together into one batch on the stove (continuous batching) — the burner (GPU) stays fully loaded instead of idling for one order; second, it reuses stock and base sauces prepped ahead of time (the prefix cache) — many orders start from the same base, so prep it once and later orders just take it. SGLang does exactly this: batch and reuse, so one expensive GPU serves as many requests as possible.

What problem does it actually solve

Training a big model is hard, but once trained, the user-facing phase is inference serving: load the weights into GPU memory, take a user prompt, generate the answer one token at a time, and return it. That sounds simple, but the difficulty is all about scale and efficiency — an H100 GPU is expensive, and generation is an inherently serial, autoregressive, token-by-token process that is "slow and waity" by nature. Done clumsily, the GPU spends huge stretches idle: either waiting for the CPU to prepare the next batch, or recomputing attention over repeated prefixes again and again. The whole value of a serving engine is to squeeze out that waste.

SGLang's answer boils down to three moves: continuous batching keeps the GPU fully loaded, RadixAttention prefix caching computes a shared opening only once, and the zero-overhead overlap scheduler hides CPU scheduling cost behind GPU compute time. As a result it can sustain higher concurrency at lower cost, generating trillions of tokens a day in production. It is also hardware-agnostic: from a single GPU to large clusters, across NVIDIA, AMD, Google TPU, Ascend NPU, and even CPUs.

Why does "serving" deserve a dedicated engine? Because inference has a very different cost structure from training. Training is a one-off offline mega-computation, whereas serving is always on: as long as the product is in use, requests keep streaming in, and every millisecond of latency and every percent of GPU utilization gets multiplied by enormous request volume into real money. So "use one card to the absolute limit" is no longer a nice-to-have but the dividing line of whether you can be profitable. SGLang is widely adopted by xAI, Cursor, and major cloud providers, running on more than 400,000 GPUs, precisely because it pushes efficiency to an industry benchmark in this demanding "online, concurrent, token-by-token" setting — it has become one of the de facto standards among open-source inference engines.

🌍 Big picture
In one line: SGLang = a frontend language (how you "express" an LLM program) + a runtime engine (how it "executes" fast and cheaply). The frontend python/sglang/lang/ offers gen / fork / join primitives so multi-step reasoning, parallel branches and structured output read like ordinary code; the runtime python/sglang/srt/ (srt = SGLang RunTime) is the real engine, pushing each request through tokenize → schedule → model forward → sample → detokenize. It is fast thanks to three weapons: RadixAttention, the zero-overhead overlap scheduler, and continuous batching.

What each half owns: frontend DSL vs runtime engine

Beginners most often ask "which half does the name SGLang refer to?" The answer is both, with a very clean split: the frontend handles "how you say it" — expressing complex logic that may include multiple model calls, branches and parallelism through compact primitives; the runtime handles "how it gets done" — executing those requests efficiently on the GPU. The frontend is a convenience layer for app authors; the runtime is the star of this guide and the source of the performance.

Frontend language DSL (lang/)

Orchestrate LLMs like a program: gen generates a span, fork spawns parallel branches, join merges results, plus structured output and multi-turn control. It makes "multi-step, parallel, constrained" prompt logic readable and reusable. It asks "how do I write this LLM program?"

Runtime engine (srt/)

The real serving engine: it receives requests and does tokenize, schedule, model forward, sample, detokenize, with RadixAttention, continuous batching, paged attention, and tensor/pipeline/expert parallelism built in. It asks "how do I execute requests fast and cheaply?" From Lesson 2 on, this guide mostly digs into this half.

Keep this throughline: the frontend "expresses", the backend "executes". You can use the runtime alone (hit the HTTP endpoint, or use the Engine class) without writing a line of frontend DSL; but when combined, the frontend's parallel primitives make it easier for the runtime to spot batchable, cacheable opportunities.

A quick intuition: suppose you want the model to critique the same passage from three angles and then summarize. In plain code you would fire three serial requests and stitch them yourself; with the frontend DSL you fork three parallel branches, each writing a gen, then join them. The key is that all three branches share the same opening (the same source text and system prompt), so the runtime can instantly see "this prefix only needs to be computed once" and RadixAttention hits the cache. The frontend says the parallel structure explicitly, and the backend can then batch and cache the work — that is the value of the two halves working together.

It helps to disambiguate a few often-confused roles: TokenizerManager and DetokenizerManager are the "translators" on the way in and out, converting between text and token ids; the Scheduler is the decision center that only decides how to batch and manage cache but does not compute itself; the one doing the matrix multiplies on the GPU is ModelRunner; and the Sampler "picks the word from the probabilities". Once you separate "who decides, who executes, who translates", no later path will confuse you.

Life of a request: a five-step pipeline

Turn the layer diagram sideways and you get a request's pipeline from in to out. We only build intuition here; Lesson 3 zooms into each station to see what a generate request actually goes through.

Cut the user's text prompt into a token-id sequence, validate params, register it as a pending request.

2

Schedule

Scheduler

Merge multiple requests into one batch, reuse shared-prefix KV cache, decide which tokens to forward this step.

3

Forward

ModelRunner

Run one model forward on the GPU to get each request's "next token" probability distribution (logits).

4

Sample

Sampler

Pick the next token by the sampling policy; unfinished requests append the new token and loop back to step 2.

5

Detokenize

DetokenizerManager

Turn token ids back into text and stream it back piece by piece until generation ends.

Note that steps 2-4 loop many times: autoregressive generation is a repeated "compute one token, append it, compute the next". The elegance of continuous batching is that each step lets new requests "cut in" to the current batch while finished ones immediately yield their slot, so the GPU almost never idles. There is also an often-missed detail: generation has two phases. The first step feeds the whole prompt in at once and fills its KV cache — called prefill; afterwards tokens pop out one by one, each step computing only the newest one — called decode. Prefill is compute-bound, decode is memory-bound; their characteristics differ, and so do the scheduling strategies — SGLang can even split them across different machines (prefill-decode disaggregation). Why it is fast: three weapons against three bottlenecks

"High performance" is not a slogan; it is one strong remedy aimed at each of three concrete bottlenecks in inference serving. The table pairs each weapon with the waste it kills, and each item is expanded in a dedicated later lesson:

Core featureBottleneck it solvesOne-line principleDetail
RadixAttentionrepeated prefixes recomputedorganize shared-prefix KV cache in a radix tree; a shared opening is computed onceLesson 7
Zero-overhead overlap schedulerGPU idles waiting on CPUwhile the GPU runs the current batch, the CPU is already preparing the next; the two overlapLesson 21

Stacked together, these four are where SGLang's "fast and cheap" comes from: compute less (prefix reuse), never idle (overlap + continuous batching), occupy less (paging + quantization). They are not isolated tricks but a set of interlocking designs around one goal — spend every moment of every expensive GPU on what matters.

Crucially, these optimizations are largely transparent to the user: you still write that one generate line and never hand-manage batches, caches, or memory. Behind the scenes the engine automatically batches mergeable requests, caches reusable prefixes, and staggers overlappable work. That is exactly what "engine" means — it encapsulates the hard performance engineering so the upper layer only expresses intent. Every later lesson is essentially lifting the lid on one layer to see what that "automatic" really does.

Hands-on: the smallest Engine usage

Skipping the HTTP server, SGLang also exposes a directly usable Engine class — a few lines do offline batch inference. Note this shows usage; what this guide really does is take you behind Engine to see what these few calls actually trigger inside the engine.

python/sglang/srt/entrypoints/engine.py ::Engineoffline batch inference
import sglang as sgl

llm = sgl.Engine(model_path="qwen/qwen2.5-0.5b-instruct")   # load model, spawn subprocesses

prompts = ["Hello, my name is", "The capital of France is"]
params  = {"temperature": 0.8, "top_p": 0.95}

outputs = llm.generate(prompts, params)               # feed a batch; the engine batches automatically
for p, o in zip(prompts, outputs):
    print(p, o["text"])

llm.shutdown()

The Engine source comment spells out its internals clearly: an engine is built from three components — TokenizerManager (tokenizes and sends requests to the scheduler), a Scheduler in a subprocess (receives requests, forms batches, forwards them, sends output tokens to the detokenizer), and a DetokenizerManager in a subprocess (detokenizes and sends results back). The processes talk over ZMQ IPC. That is exactly the layer diagram and pipeline above, in code. A live deployment is started by python/sglang/launch_server.py, which wraps this engine in an HTTP server exposing OpenAI-compatible APIs and a native /generate endpoint.

Why split into multiple subprocesses talking over IPC, instead of one big loop? Because tokenize and detokenize are CPU string processing while the model forward is heavy GPU compute; putting them in separate processes lets the CPU work and the GPU work truly pipeline in parallel without blocking each other — back to the "zero-overhead overlap" idea. You need not memorize every component yet; just form this mental map: one generate call drops in, and behind it a small, clearly divided team pipelines the work. 📌 Key points
  • Role: SGLang is an inference serving engine — it solves "how to put a trained model online fast and cheaply"; it does not train, and it is not a database.
  • Two halves: frontend DSL (lang/, gen/fork/join to express LLM programs) + runtime engine (srt/, the real executor). Frontend expresses, backend executes.
  • Pipeline: a request goes tokenize → schedule → forward → sample → detokenize; the middle three loop until generation ends.
  • Why fast: RadixAttention (prefix reuse) + zero-overhead overlap scheduler (no idling) + continuous batching (full batches) + paging/quantization (memory savings).
  • Cross-hardware: upper layers are hardware-agnostic; only the bottom operator kernels change per hardware — single GPU to large clusters across NVIDIA/AMD/TPU/NPU/CPU.

The project map

Last lesson we sketched SGLang from 10,000 feet: it is an inference serving engine made of two halves, a frontend language and a runtime engine. 🔌 Analogy
Picture SGLang as a working office building. The front desk on floor one (TokenizerManager) greets visitors and turns spoken requests into standard work orders; the dispatch room on floor two (Scheduler) is the brain that decides which orders to merge into one batch, who goes first, and how to queue — but the dispatcher does no hands-on work; the ones actually turning bolts on the line are the technicians (TpWorker / ModelRunner) who tend the expensive machines in the machine room (GPUs and operator kernels); when a job is done, shipping & packing (DetokenizerManager) translates the half-product into a finished item the customer can read and mails it back. The key: front desk, dispatch+floor, and shipping are three independent departments (processes) that pass work orders via the internal mail system (ZMQ), so none blocks another — that is how reception and the shop floor run at the same time.

The whole repo can first be split crudely into two halves plus a foundation: python/sglang/lang/ is the frontend DSL (how you "express" an LLM program), python/sglang/srt/ is the runtime engine (srt = SGLang RunTime, how it "executes" fast and cheaply); underneath both sit sgl-kernel/ (ahead-of-time compiled C++/CUDA ops) and python/sglang/jit_kernel/ (lightweight just-in-time kernels). The repo's two halves: lang/ and srt/

Beginners opening the repo get scared by dozens of directories, but one first cut into two halves makes it clear: one half teaches you how to write an LLM program, the other runs requests fast and cheaply. The former is a convenience layer for app authors; the latter is the source of performance and what we dig into. The name SGLang refers to both halves, but their code size, complexity, and distance from the hardware differ entirely.

Frontend DSL · python/sglang/lang/

Offers gen (generate a span), fork (spawn parallel branches), join (merge results), plus system/user/assistant roles and structured output. It makes "multi-step, parallel, constrained" prompt logic read like an ordinary program. Key files: api.py, ir.py, interpreter.py. It asks "how do I write this LLM program?"

Runtime engine · python/sglang/srt/

The real serving engine: it receives requests and does tokenize → schedule → model forward → sample → detokenize, with RadixAttention, continuous batching, paged attention, and tensor/pipeline/expert parallelism built in. It asks "how do I execute requests fast and cheaply?" From this lesson on we mostly live in this half; it is also the thickest stack of code in the repo.

Keep this dividing line: lang/ expresses, srt/ executes. You can use srt/ alone (hit the HTTP endpoint or use the Engine class) without writing any frontend; conversely, when the frontend says the parallel structure explicitly, it helps the backend spot batchable, cacheable opportunities — exactly the logic behind Lesson 1's "critique the same passage from three angles" example.

Why bind a "language" and a "runtime" in one repo? Because they are two ends of the same thing: the more clearly the frontend can express "which calls can run in parallel, which prefixes are shared", the more chances the backend has to cash those structures into real speedups. Many other frameworks do only the runtime and leave the frontend for users to hand-stitch, so the scheduler receives a pile of isolated requests "with no visible relationships" and misses batching and caching opportunities. SGLang puts both ends together precisely so the information between "express" and "execute" is not lost in transit. That is also why the Lang in the name is not just "language" but "the kind of language the runtime can understand" — a design philosophy that sets it apart from a plain inference server. For a beginner, though, remembering that srt/ is the star is enough: the frontend is icing, while the runtime is this engine's real engine bay, and the object this guide takes apart layer by layer.

The three-process model: who lives in which process, talking how

The stack above is the static code structure; but once the engine runs, it is not one big loop but forks into several processes working in parallel. This is the most overlooked yet most crucial point of SGLang's architecture: tokenize, schedule+forward, and detokenize live in different processes, passing messages over ZMQ IPC. Why split this way? Because tokenize/detokenize are CPU string processing while the model forward is heavy GPU compute; putting them in separate processes lets CPU work and GPU work truly pipeline in parallel without blocking — the engineering basis for "zero-overhead overlap" (Lesson 21).

Never confuse the three roles: TokenizerManager is in the main process (so are the HTTP server and the Engine); Scheduler is in a subprocess, and tensor parallelism (TP) starts one Scheduler process per rank, each holding its own TpWorker and ModelRunner and occupying one GPU to do the forward; DetokenizerManager is yet another subprocess. Remember: the Scheduler is the process holding the GPU, while the two Managers are just "translators" on the way in and out — do not lump them together.

This "who lives in which process, how they get launched" logic is spelled out plainly in the source. The snippet below is the skeleton of process forking at engine startup — it first allocates ZMQ ports for all three, then launches the Scheduler subprocess and the Detokenizer subprocess, and finally leaves the TokenizerManager in the main process:

python/sglang/srt/entrypoints/engine.py ::_launch_subprocessesthe three-process fork point
def _launch_subprocesses(cls, server_args, ...):
    # 1) allocate IPC ports for all three (ZMQ communication)
    port_args = PortArgs.init_new(server_args)

    # 2) Scheduler subprocess: one per TP rank, holds TpWorker/ModelRunner, does GPU forward
    scheduler_init_result, scheduler_procs = cls._launch_scheduler_processes(
        server_args, port_args, run_scheduler_process_func)

    # 3) DetokenizerManager subprocess: detokenize, send results back
    detoken_procs, _ = cls._launch_detokenizer_subprocesses(
        server_args=server_args, port_args=port_args, ...)

    # 4) TokenizerManager stays in the main process: tokenize, send/recv
    tokenizer_manager, template_manager = init_tokenizer_manager_func(
        server_args, port_args)

    scheduler_init_result.wait_for_ready()   # wait for the model to load in the subprocess
    return tokenizer_manager, template_manager, port_args, scheduler_init_result, ...

The source comment spells out the three components: TokenizerManager tokenizes and sends requests to the scheduler; the Scheduler in a subprocess receives requests, forms batches, forwards them, and sends output tokens to the detokenizer; the DetokenizerManager in a subprocess detokenizes and sends results back. The HTTP server, Engine, and TokenizerManager all run in the main process, and processes talk strictly over ZMQ IPC (a port per process). A live deployment is wrapped in an HTTP server by launch_server.py; and every component's "switches and knobs" — model path, tensor-parallel size, memory fraction, max concurrency — live in ServerArgs in srt/server_args.py, the page you will most often consult when tuning.

Why use ZMQ inter-process communication rather than writing all three as threads inside one process? At root it is the reality of Python and the GPU: Python's global interpreter lock (GIL) makes it hard for CPU tokenization and GPU forward to truly run in parallel as threads; split into separate processes, each with its own interpreter, tokenize, forward and detokenize can advance physically at the same time without fighting over one lock. ZMQ then provides a lightweight, cross-process message queue: each process binds a port, and requests and results flow between them like parcels on a conveyor, the sender not waiting for the receiver to finish — this asynchronous decoupling is the physical basis of "zero-overhead overlap scheduling". The cost is a few extra ports and subprocesses at startup, plus a watchdog keeping an eye out: if a subprocess crashes, the main process notices and exits promptly instead of hanging forever. Once you understand "why split processes, why a message queue", you have grasped the most counterintuitive and most crucial piece of SGLang's architecture.

Life of a request

The map is drawn; this lesson brings it to life: we follow a real generate request from the moment you hit enter, watching it cross three processes, get merged into a batch, compute one token after another on the GPU, then turn back into text streamed home to you. This is a "10,000-foot" end-to-end trace — each station's internals are left to later dedicated lessons; here we just string the whole path into one line, so you carry a replayable mental animation of a "request's life".

🔌 Analogy
Think of one request as a table's worth of dishes ordered at a restaurant. You state your needs at the counter (TokenizerManager) and the server writes a standard menu ticket; the ticket goes into the kitchen dispatch (Scheduler), where the chef does not cook table by table but merges tickets that arrive together into one batch; each time the stove (GPU / ModelRunner) "fires once", every table in the batch advances one step; dishes are not served all at once but course by course (streaming output) — exactly how autoregressive generation "pops one word at a time". The whole meal has two phases: the opening prep and base stock (prefill, reading the entire prompt in at once) takes the most effort, after which serving course by course (decode, computing only the newest word each time) is a repeated quick small step.
🌍 Big picture
A generate's whole journey is just last lesson's "three-process diagram" actually walked once: HTTP entry → TokenizerManager tokenize → (ZMQ) → Scheduler batch → ModelRunner forward → Sampler sample → DetokenizerManager detokenize → (ZMQ) → streaming SSE back. The crucial insight: the big middle stretch loops many times — one prefill reads the prompt in, then many decodes pop out word by word, and continuous batching "cuts in / yields" on every turn so the GPU almost never idles.

Walk it through: your POST /generate first lands in the handler in entrypoints/http_server.py, which wraps the payload into a GenerateReqInput and hands it to TokenizerManager.generate_request. In the main process, TokenizerManager tokenizes the text into token ids, normalizes params, packs a TokenizedGenerateReqInput (defined in managers/io_struct.py), and sends it over ZMQ to the Scheduler subprocess. The Scheduler receives, queues, batches, forwards and samples on the GPU, and sends newly generated tokens to the Detokenizer subprocess; the latter does incremental detokenization, sends "the few new characters this time" back to the main process, and the HTTP layer pushes them to you piece by piece as SSE (text/event-stream), until a stop token or the length limit.

Here is an easily overlooked but important design: detokenization must be "incremental". Each token the model emits does not necessarily map to a complete character or word — some characters take several tokens to form. Re-detokenizing the entire generated text from scratch on every new token would be wasteful and could wrongly slice a half character. So the DetokenizerManager remembers "how many characters were already sent" and each time pushes only the small slice that is genuinely new and safe to display (called sent_offset in the source). That is the real source of answers appearing "word by word" in your UI: not the model finishing then returning, but the engine streaming text incrementally as it computes. Streaming improves experience so much precisely because the user starts seeing the answer the instant the first token is computed, instead of waiting for the whole generation to finish — especially crucial for long answers of hundreds of tokens.

Compress the whole path into one mnemonic: "main tokenizes, subprocess computes, subprocess translates, main emits". A request enters and is tokenized and registered in the main process, crosses one ZMQ hop to the Scheduler subprocess to be batched, forwarded and sampled, the new token crosses another ZMQ hop to the Detokenizer subprocess to be turned into text, and finally loops back to the main process to stream out as SSE. Note these four stages are not synchronously blocking — while the Scheduler forwards token N, the Detokenizer may still be translating token N-1, and the main process has already pushed token N-2 to the user; the three processes run different relay legs at the same time. It is exactly this cross-process pipeline that makes "a request's life" look like a straight line at the macro level yet, microscopically, the result of three parties collaborating in parallel. Carrying this end-to-end animation, we are ready to dive into any stage and see its internals.

The scheduler loop: how requests get batched and run

Across the whole path, the thing most worth seeing first is the Scheduler brain — because it is a never-stopping loop. It does not "compute one as each arrives" but repeatedly does four things: receive requests → form the next batch → run that batch on the GPU → process results. These four are the engine's heartbeat; here they are unrolled as vertical steps:

1

recv_requests

This loop is astonishingly short in the source — below is the "normal mode" event-loop skeleton, mapping almost one-to-one to the four steps above (there is also an event_loop_overlap variant that lets the GPU run the current batch while the CPU prepares the next, expanded in Lesson 21):

python/sglang/srt/managers/scheduler.py ::event_loop_normalthe engine's heartbeat
def event_loop_normal(self):
    """A normal scheduler loop."""
    while True:
        # 1) receive requests (via ZMQ from TokenizerManager), enqueue
        recv_reqs = self.request_receiver.recv_requests()
        self.process_input_requests(recv_reqs)

        # 2) form the next batch: prefill or decode this step?
        batch = self.get_next_batch_to_run()
        self.cur_batch = batch

        if batch:
            # 3) forward + sample on the GPU
            result = self.run_batch(batch)
            # 4) process results: send new tokens to the detokenizer
            self.process_batch_result(batch, result)
        else:
            self.on_idle()

        self.last_batch = batch

Here is the secret entrance to SGLang's throughput: step 2's get_next_batch_to_run re-forms the batch every turn, so continuous batching "cuts in" right here — a new request can squeeze into the current batch any turn, a finished one yields any turn (Lesson 5); RadixAttention computes a shared prefix's KV cache only once (Lesson 7); the concrete batching/queuing policy is Lesson 20, and a close source read of the whole loop is Lesson 18.

One more key detail: step 3's run_batch is the only place that actually touches the GPU. It hands the whole batch to ModelRunner.forward inside the TpWorker, runs one model forward on the card to get each request's "next token" logits, then passes them to the Sampler to sample by temperature / top-p / top-k. Note the whole loop runs single-threaded, turn by turn — no callbacks flying around, no complex multi-thread synchronization, just a plain while True. This "one big loop governs everything" design looks simple but is exactly why it is easy to understand, debug and optimize: watch this one loop and you can answer "what is the engine doing now, what will it do next". The overlap variant event_loop_overlap is faster because it staggers step 3 "run this batch" and step 4 "process the last batch's result" into a pipeline, so the CPU prepares while the GPU computes — but the skeleton is the same loop; once you get the normal version, overlap is just its "accelerated variant".

The decode loop: why it goes around so many times

The biggest misconception about "one request" is that the model "writes the whole answer in one go". In fact generation is token-by-token, autoregressive: compute one token, append it to the input, compute the next, and so on until the model emits a stop token or hits the max length. So steps 2–4 loop tens to hundreds of times, advancing one word per turn:

Because the batch is re-formed every turn, continuous batching gets its chance to "cut in": a new request need not wait for the current batch to finish — it can join next turn; a finished request immediately yields its slot in the batch, and the freed compute goes to others at once. That is why SGLang keeps GPU utilization so high — every step of the loop is a chance to reshuffle, rather than "run one batch to the bitter end".

One trap beginners hit: this "batch" is not the same as a training "batch". In training, samples in a batch are often length-aligned and go in and out together; in inference serving, each request arrives at a different time and generates a different length — one just sent in and needs prefill, one has emitted dozens of tokens and is decoding, one is about to finish. The essence of continuous batching (also called in-flight batching) is to not force everyone in lockstep: each step dynamically assembles a batch from whoever is present, adding arrivals and dropping departures. That is also why the scheduler loop must be fast enough — its per-turn overhead gets multiplied by "loops many times", which is exactly why the zero-overhead overlap scheduler (Lesson 21) hides CPU batching overhead behind GPU compute. Stack "token-by-token loop" and "re-batch every turn" together and you see that almost the entire story of SGLang's throughput and latency lives in this loop.

Prefill vs decode: two different jobs

The loop actually mixes two completely different kinds of step, and telling them apart is the prerequisite for every later performance lesson. Prefill happens when a request just arrives: read the entire prompt in at once and fill its KV cache — a prompt has hundreds to thousands of tokens computed in parallel as a big slab, so it is compute-bound, but usually done only once. Decode is every later turn computing only the single newest token; the compute is tiny, the bottleneck is repeatedly moving the huge KV cache, so it is memory-bound and loops many times. But in the plainest single-machine case, just remember this throughline: one request = one prefill + many decodes, and continuous batching lets countless such requests pipeline out of phase on the same card. That is the entire skeleton of a "request's life" — every later lesson adds flesh to one joint of this skeleton.

Why insist on telling the two phases apart? Because they decide the direction of effort for almost every later optimization. Prefill is compute-bound, so optimizing it works on compute: more efficient attention kernels, better parallel partitioning; decode is memory-bound, so optimizing it works on memory: how the KV cache is laid out, paged, compressed, and reused across shared prefixes (the home turf of KV cache in Lessons 4 and 29–32, paged attention in Lesson 6, and RadixAttention in Lesson 7). A common intuition trap is "thinking all the cost is in thinking up the answer", when in fact for long conversations repeatedly moving an ever-growing KV cache is the real bulk of decode-phase cost. So when you later meet all kinds of "save memory, lift bandwidth" tricks, return to this prefill–decode timeline and you will see exactly which phase they are saving. Carve this throughline into your mind: prefill once, decode hundreds of times; prefill races on compute, decode races on memory — it is the key to understanding all of Part 2.

Autoregressive decode & the KV cache

Part 1 gave you the bird's-eye view of SGLang. From here we fill in the "why these designs exist" foundations. The first cornerstone underlies nearly every LLM-inference optimization: the KV cache. To explain it we first need to see how a model emits text one token at a time: once you understand autoregressive generation, you'll see the KV cache isn't a nice-to-have — it's what makes generation go from "possible" to "practical."

🔌 Analogy
When you do a long piece of long-hand multiplication, you write the partial products on scratch paper and reuse them on the next step, instead of multiplying from scratch again. The KV cache is the model's scratch paper: it stores the intermediate results (each word's Key/Value) so every new word reuses history and never recomputes it. Throw the scratch paper away (no cache) and writing each new letter means redoing all the previous ones — too slow to be usable.
🌍 The big picture
An LLM is autoregressive: each new token must look back at all history. Recomputing the whole sequence every step is prohibitively expensive; the KV cache stores each token's read-only Key/Value once and reuses it, squeezing each step from "recompute everything" to "just the new word's attention over history". The price is HBM — the cache grows linearly with context, the very thing paging, prefix reuse and quantization later fight.

Autoregressive generation: one token at a time

An LLM is autoregressive: it predicts only the next token, appends it to the input, predicts the next, and repeats until it emits an end token or hits a length limit. So the model doesn't "think the whole sentence up at once" — it loops one token at a time, and every step looks back at everything it has written so far.

1

Feed the current sequence

Input = "prompt + already-generated tokens".

2

One forward pass

The model outputs a probability distribution for the next token (logits).

3

Sample one token

Pick the next token by temperature/top-p, etc.

4

Append, loop

Append the new token to the sequence; back to step 1 — until done.

The crux is attention: each new token must "greet" (attend to) all previous tokens to know what to say in context. Which raises the question — that history is needed every single step, so do we recompute it every step?

One more essential point: autoregression is inherently serial — token N can't start until token N−1 has actually been sampled and appended. This is unlike training, where a whole known text is computed in parallel (a causal mask scores all positions at once). The serial nature of inference means you cannot amortize a single request's latency by "computing a few more tokens"; one request's speed is hard-capped by the per-token loop. Since a single request can't be sped up, the only engineering way out is to "pack more requests side by side and share each step's fixed cost" — which lands squarely on the doorstep of batching and caching, the intuition this whole Part 2 is built to give you.

Without a cache: O(t²) of repeated work

Attention has each token produce three things: Q (Query — what I'm looking for), K (Key — the label I can be found by), V (Value — the information I carry). A new token uses its Q against all historical tokens' K to decide "whom to attend to", then sums their V.

❌ Naive (no cache)

For every new token, re-run the whole sequence from scratch: recompute every historical token's Q/K/V and all the attention among them. Step t costs O(t²), and the historical K/V come out identical every time — pure waste.

✅ With a cache

Historical tokens' K/V are fixed the moment they were first generated. Store them; a new token computes only its own Q/K/V, then attends its Q against the cached K/V. Each step is just O(t), dropping all the repeated work.

In one line: historical K/V are "read-only", computed once and reused. Caching them squeezes each step from "recompute the whole sequence" to "just the new word's attention over history" — that is the KV cache.

Follow one decode step: what happens inside the cache

Abstract big-O is less convincing than a concrete trace. Let the prompt be "The capital of France is" and the model continue with "Paris". In prefill, the model computes all prompt tokens at once, so the cache holds 6 rows of K/V and emits the 1st output token "Par". Then step-by-step decode begins, each step doing the same three actions — append a row, read everything, predict the next:

decode, cache step by step: each step adds one K/V row but attends to all history (the cache only grows, never changes)
prefillThecapitalofFranceis→cache 5 rows, output "Par"
decode①…5 rows…Par→append 1 = 6 rows · "Par" Q reads 6 rows → "is"
decode②…6 rows…is→append 1 = 7 rows · "is" Q reads 7 rows → <eos>

Notice two details. First, each step computes only "one new token", yet it attends to all of the cached history — so as generation grows, each step reads an ever-larger cache while doing tiny compute, spending its time "moving the cache from HBM into the compute units". That is the root of decode being memory-bound, and why the key to decode throughput isn't FLOPs but memory bandwidth and batch size. Second, the cache is append-only, never modified — once a history row is written it never moves. It is exactly this "read-only, shareable" property that lets later requests share the same prefix's KV (Lesson 7, RadixAttention). In other words, the "cache" here is not just a one-off optimization; its data-structure properties directly decide how many tricks the whole engine can pull off downstream.

An often-missed detail: GQA makes KV heads far fewer than Q heads

You may have noticed the ledger said "8 KV heads" while such models have 32 attention heads. That's Grouped-Query Attention (GQA): many Q heads share one group of K/V heads. It barely affects quality yet shrinks the KV cache to about 1/4. Why call it out in a foundations lesson? Because the KV cache is the HBM dominator, and GQA is the architectural way to slim it — complementary to SGLang's engineering HBM tricks (paging, prefix reuse, quantization): one changes "how big each token is", the others change "how to lay it out, how many copies, how many bits". When you read the memory-management lessons, remember these two lines advance in parallel and stack.

How it's stored: a "pool" in SGLang

In the SGLang runtime, all tokens' K/V don't hang off individual requests — they go into one big HBM pool. For ordinary multi-head attention that pool is managed by MHATokenToKVPool: at startup it carves out two big buffers for K and V sized by "how many tokens × layers × KV heads × head_dim", and every new token writes one row. Making the cache a "pool + per-row allocation" is the physical basis for the tricks in paging (Lesson 6) and prefix reuse (Lesson 7).

A few hundred concurrent requests and the KV cache alone eats tens to hundreds of GB. That is why "save HBM" is an inference engine's number-one job — paging, prefix reuse, quantization and KV-cache quantization all fight over this ledger.

There's a causal layer that's easy to miss: the cache's total capacity directly caps how many requests you can serve at once. The HBM pool is fixed, so the total number of tokens it can hold is fixed; every request consumes "context length × bytes per token" of cache. Thus longer context means lower concurrency — the two contend for the same HBM. So the KV cache isn't just a performance issue but a hard capacity-vs-concurrency constraint: the SGLang scheduler (from Lesson 18) must answer "is there enough HBM to admit one more request?" every step, and how boldly it can answer depends on how frugally the cache is managed. Keep this chain in mind: save a bit of cache → admit a few more requests → higher whole-machine throughput. That is exactly why we spend a whole "memory management" part (Lessons 29–32) digging into it.

Precisely because decode is memory-bound (its compute is tiny; the time goes to moving the KV cache and weights from HBM into the compute units), batching many requests together pays off — one weight read serves many requests so the GPU stops idling on a single one. Autoregression + the KV cache define inference's two fundamentals: a serial per-token loop (which sets the floor on single-request latency) and an append-only cache that grows linearly with context (which sets the ceiling on concurrency). The next four lessons are all moves on these two fundamentals — Lesson 5 uses continuous batching to turn "serial single" into "parallel many", reclaiming the FLOPs wasted during decode; Lesson 6 uses paging to pack that linearly-growing cache tighter with less fragmentation; Lesson 7 uses RadixAttention to let different requests share a prefix's cache, cashing the "read-only" property into real compute and memory savings; Lesson 8 puts it all on the throughput-vs-latency scale and tells you which end each trick pushes on. Read on with three rulers — "cache = money, concurrency = throughput, serial = latency floor" — and you'll see at a glance what every later design saves and trades.

  • KV cache: historical K/V are read-only, so cache and reuse them; each step drops from re-running the whole sequence O(t²) to O(t).
  • HBM dominator: cache ≈ 2×layers×KV-heads×head_dim×bytes per token, growing with context length — saving HBM is job #1.
  • Two phases: prefill is compute-bound (fills once), decode is memory-bound (appends a row); one request = 1 prefill + N decodes.
  • Bridges: memory-bound decode ⇒ batching pays off (Lesson 5); pool/row-managed cache ⇒ paging (Lesson 6) and prefix reuse (Lesson 7) become possible.

Continuous batching

Last lesson nailed one iron law: decode is memory-bound — each step computes just one new token while the compute units idle, with time spent moving weights and the KV cache out of HBM. That seeds this lesson's core motivation: since a single request can't be sped up, pack more requests together so one "weight read" serves as many requests as possible. How you pack them decides whether the GPU sprints at full load or idles — and that is exactly what continuous batching solves, the first cornerstone of SGLang's high throughput.

🔌 Analogy
Picture two buses. A charter bus won't leave until it's full and makes no stops — everyone must ride all the way to the terminus. Even if your stop is third, you ride the whole route; passengers who arrived early just sit in seats that produce nothing. A hop-on/hop-off shuttle stops at every stop: whoever has arrived gets off and frees a seat, and whoever is waiting hops on instantly. The bus stays always full — no empty seats, no idle waiting. Static batching is the charter bus; continuous batching is the shuttle — and the shuttle pushes that one bus (the GPU) to its utilization limit.
🌍 The big picture
In one line: continuous batching = the scheduler re-forms the batch at every decode step. A finished request leaves on the spot and frees its KV slot immediately; a waiting request joins on the very next step — so the batch stays full and the GPU never idles on any single request. It's also called iteration-level batching or in-flight batching. In SGLang, the scheduler loop calls get_next_batch_to_run() each round to "form the batch for this step."

Static batching: waiting on the slowest one

The naive approach is static (padded) batching: gather a few requests that arrive together into a batch, push them through the model, and run them start to finish as a group. It sounds reasonable, but it has three fatal wastes — all rooted in "a batch of requests being chained together."

First, the whole batch waits for the slowest. Within one batch, some requests hit their end token after 10 tokens, others need 500. But static batching only finishes when the longest one does. Requests that finished early just ride along, holding their slot while producing nothing. Second, padding waste: finished sequences are meaningless, yet their slots tag along through every forward pass — pure wasted compute. Third, new requests wait for the whole batch to drain: even if the GPU has free capacity right now, an arriving request can't enter until the current batch fully ends. The result: GPU utilization is dragged down — the later the batch, the fewer requests are still doing real work.

Continuous batching: re-form the batch every step

The core action is one sentence: don't form the batch once and freeze it — re-assemble the batch at every decode step. Think of generation as a never-ending loop; each round the scheduler does four things — evict the finished, admit the waiting, form a new batch, forward one step:

1

Evict finished requests

Whoever emitted an end token or hit the length limit last step is removed from the batch, freeing its KV-cache slot immediately.

2

Admit waiting requests

Check whether HBM allows it and which the policy lets in first, then fill the freed slots from the queue (running its prefill first if needed).

3

Form this step's batch

Assemble all currently-running requests into one batch — do prefill first if there's prefill work, else a decode batch.

4

Forward one step, loop

Forward the whole batch one token, update each KV cache, back to step 1 — the batch stays full and flowing.

The key insight is that a finished request can leave mid-flight, instantly. Why can it leave — and free its HBM — right away? Because, as last lesson showed, each request's KV cache is managed independently by row/slot; one request's K/V is not entangled with another's. So the moment a request ends, its KV slots can be reclaimed in place and handed to the next queued request — no waiting for the whole batch, no reshuffling. Why it wins: back to "memory-bound"

Continuous batching is a throughput weapon, and the root reason traces back to Lesson 4. decode is memory-bound: to produce one token you read the entire model's weights from HBM, yet the actual matmul serves just one token — compute is badly underused. And reading the weights costs about the same whether the batch holds 1 request or 100 — one weight read can forward a whole batch at once. So stacking requests together amortizes that fixed "read the weights" cost across every request: the bigger the batch, the thinner the amortization, the more useful tokens per unit of compute.

Static batching wants this dividend too, but it can't keep the batch full: as requests finish one by one, the batch's effective size shrinks, the amortization decays fast, and the back half practically idles over empty slots. The entire value of continuous batching is refilling the batch every step — making "read weights once, serve a big batch" happen continuously, end to end. Intuitively: on the same GPU, static batching might do real work only ~60–70% of the time, while continuous batching approaches full load. That's why it's widely seen as the single most important throughput technique in modern inference engines — it changes no model and loses no accuracy, lifting utilization purely by how it batches.

In SGLang: calling get_next_batch_to_run every round

In SGLang's runtime, the scheduler (its event loop is detailed in Lesson 18) calls get_next_batch_to_run() once per iteration to "form the batch for this step." The method first filters out requests that finished last step and merges newly arrived prefill into the running batch, then makes a key decision: is there prefill work? If so, return a prefill batch; otherwise, build a decode batch. That one "decide the batch afresh each step" is exactly how continuous batching lands in code.

python/sglang/srt/managers/scheduler.py ::Scheduler.get_next_batch_to_runform this step's batch every schedule round
def get_next_batch_to_run(self) -> Optional[ScheduleBatch]:
    # …first filter out last step's finished reqs, merge prefill into running_batch…

    # Try to form a prefill batch first (do new prompts' prefill if any)
    new_batch = self.get_new_batch_prefill()

    if new_batch is not None:
        # Run prefill first if possible
        ret = new_batch
    else:
        # Otherwise build a decode batch from the currently running requests
        self.running_batch = self.update_running_batch(self.running_batch)
        ret = self.running_batch if not self.running_batch.is_empty() else None
    return ret

Read this and you grasp the soul of continuous batching: the batch is not a static object but a result recomputed every step. Finished requests leave during the filter phase and free their KV slots; waiting requests are admitted in the get_new_batch_prefill phase — each loop refreshes the batch. Who joins, and how many is decided by the schedule policy (Lesson 20), and the prefill-vs-decode priority tradeoff is settled right here too.

The cost and tradeoff: no free lunch

Continuous batching maxes out utilization, but the price is more "bookkeeping" every step: filtering the finished, admitting the waiting, re-forming the batch, reclaiming and allocating KV slots one by one. This means the scheduler itself must be cheap enough — if the per-step Python scheduling overhead is too large, the GPU ends up idling while the CPU forms the batch, defeating the purpose. That motivates two key SGLang follow-ups:

Scheduling must be "zero-overhead"

Since the batch is re-formed every step, the scheduling logic is on the hot path. SGLang's zero-overhead overlap scheduler (Lesson 21) hides the CPU batch-forming cost inside the previous step's GPU compute, so the GPU never idles on scheduling — the prerequisite for actually cashing in the throughput dividend.

Long prompts mustn't clog the queue

If a request's prompt is extremely long, its prefill can hog a whole step and stall every decoding request, breaking the batch's flow. SGLang's chunked prefill (Lesson 22) slices a long prefill into chunks, interleaving them with decode steps to keep the batch smooth.

Continuous batching thus becomes the engine's first throughput engine. PagedAttention & paged KV

Last lesson kept the batch always full, but that hides an unstated premise: a finished request must free its HBM immediately, and a waiting one must get HBM immediately. The question is — how does the KV cache actually live in HBM? If you use the old way and reserve a contiguous block of HBM per request, you'll find HBM wasted by the bucketload and concurrency capped hard. This lesson's PagedAttention and paged KV borrow the operating system's decades-old "paging" wisdom for fighting fragmentation and apply it to the KV cache, so HBM packs tighter, with near-zero fragmentation.

🔌 Analogy
Picture a hotel with two booking styles. The old way: for each guest, reserve a whole contiguous floor up front no matter how many nights they stay — sized for the longest possible stay. Most guests stay two or three nights, yet the whole floor stays locked away from everyone else; worse, it must be a contiguous floor, so scattered empty rooms here and there can never add up to one. The new way: rent single rooms by the night, open as many as you need, rooms need not be adjacent, and the front desk keeps a directory: "Zhang San → rooms 301, 507, 902." A guest checks out, the room is reclaimed that night. The old way is "contiguous reservation"; the new one is "paging" — and it stuffs far more guests into the same building.
🌍 The big picture
In one line: PagedAttention cuts each request's KV cache into fixed-size "pages (blocks)" and uses a "page table" to map logical positions to physical pages, allocating on demand and freeing when done. This is exactly the OS's virtual memory / paging: a process sees a contiguous logical address space, but physically it's scattered across arbitrary page frames, with a page table tying the two together. Because the page table does the "finding," physical pages need not be contiguous, so fragmentation nearly vanishes and HBM utilization approaches 100%. Pioneered by vLLM, it's now broadly adopted by nearly every inference engine, SGLang included.

The pain: why contiguous reservation burns HBM

Back to Lesson 4: each request's KV cache grows as it generates, but you don't know in advance how many tokens it will produce. The naive approach is to reserve one contiguous block of HBM per request, sized for the max length — cap of 2048 means grabbing KV space for 2048 tokens up front. This "contiguous reservation" looks convenient but hits two classic fragmentation problems at once, wasting precious HBM.

Internal fragmentation (over-allocation)

Most requests never reach the max length: cap of 2048, but maybe only 60 tokens are generated. The remaining ~2000 slots sit empty yet locked by that request — usable by no one. The bigger the reservation, the worse the waste. That's internal fragmentation.

External fragmentation

Requests come and go, leaving free blocks of uneven sizes wedged in between. A new request wants one contiguous block, but what's left is little gaps scattered around — enough in total, yet un-assemblable into one block, so it idles. That's external fragmentation.

Stack the two and the consequence is blunt: HBM isn't even full, yet a new request no longer fits. And as Lesson 5 said, the number of concurrent requests (batch size) is the throughput lifeline — bigger batches amortize the fixed weight-read cost thinner. Contiguous reservation burns HBM on empty slots and gaps, which directly cuts how many requests you can run at once, pressing the throughput ceiling down hard. So "how you lay out KV" isn't a detail — it's what sets the concurrency limit.

The core of paging: fixed pages + a page table

PagedAttention needs just two things: fixed-size pages and a page table mapping logical to physical. Think of a request's KV as a logically contiguous token stream, but underneath it's sliced into pages of page_size tokens each, with every page drawn separately from the pool. Which page sits where in physical HBM is irrelevant — a page table records "page N → physical block id," and you follow it when you need to fetch.

1

Slice into pages

The token stream is cut by page_size (e.g. 16) into pages — logically contiguous, physically separate.

2

Allocate on demand

Each time the sequence fills a page, ask the allocator for one more free block — take only what you use, never reserve to the cap.

3

Record in the page table

Write "logical page → physical block" into this request's page table (in SGLang, the token→slot mapping).

4

Free when done

When a request ends, all its pages are reclaimed in one batch into the free pool, ready for the next request instantly.

The crux is the page table in step 3: when the attention kernel looks back at historical K/V, it doesn't fetch by contiguous address but first consults the page table, then "gathers" K/V from the corresponding physical blocks. Thanks to this indirection, physical blocks can go anywhere — they need not be contiguous, which is exactly why paging kills external fragmentation. One often-missed tradeoff: smaller page_size is not always better. Smaller pages waste less internally (at most a fraction of a page), but the page table grows longer and table lookups / kernel indirection cost more; bigger pages address more cheaply but waste more on each request's last unfilled page. Common values are 1, 16, 32 and the like, decided by model/hardware tradeoffs — the classic "fragmentation vs overhead" balance.

In SGLang: the paged allocator and req_to_token

In SGLang this mechanism is carried by a paged token→KV allocator: it manages a whole pool of KV slots but hands them out with the "page" as the minimum unit. Paired with the KV pool from Lesson 4, a req_to_token map records each request's token → physical slot — the in-code incarnation of that "page table." Below is the allocator's skeleton: alloc takes pages, free returns pages, all at page_size granularity:

python/sglang/srt/mem_cache/allocator/paged.py ::PagedTokenToKVPoolAllocatorpaged allocator: hands out KV slots by page
class PagedTokenToKVPoolAllocator(BaseTokenToKVPoolAllocator):
    # An allocator managing the indices to kv cache data; output is page-aligned
    def __init__(self, size, page_size, dtype, device, kvcache, need_sort):
        super().__init__(size, page_size, dtype, device, kvcache, need_sort)
        self.num_pages = size // page_size        # total slots / page_size = #pages

    def alloc(self, need_size):
        # page-aligned allocation: return the physical indices of some pages
        num_pages = need_size // self.page_size
        out_pages = self.free_pages[:num_pages]   # take num_pages off the free list
        self.free_pages = self.free_pages[num_pages:]
        out_indices = (out_pages[:, None] * self.page_size
                       + torch.arange(self.page_size, device=self.device)).reshape(-1)
        return out_indices                        # expand to token-level slot indices

    def free(self, free_index):
        # on finish: dedup the page ids it held and return them in one batch
        free_page_indices = torch.unique(free_index // self.page_size)
        self.free_pages = torch.cat((free_page_indices, self.free_pages))

Read this and you grasp the essence of paging: HBM is no longer reserved wholesale per "request" but retailed per "page." alloc takes some pages off free_pages, then multiplies page ids by page_size and adds the in-page offset to expand into token slot indices; free divides those indices by page_size, dedups to page ids, and returns them in one batch when the request ends. Note the class comment "output is page-aligned" — it guarantees pages stay cleanly aligned and easy to reclaim, the physical basis for continuous batching's (Lesson 5) "free on finish, fill on arrival."

Answer: C. Paging's core is the 'fixed pages + page table' indirection, just like OS virtual memory. RadixAttention & prefix caching

Last lesson's paging handed us a loaded capability: different requests' page tables can point at the same physical blocks. This lesson pushes that to the limit. Real traffic hides one constantly-overlooked fact — requests usually start out identical: the same system prompt, the same few-shot examples, the same agent scaffold, and in multi-turn chat every turn resends the entire history. Recomputing that shared opening's KV for every request and every turn is enormous waste. RadixAttention's answer: store every request's KV in a radix tree, so a shared prefix is computed once and reused automatically, across requests and across turns. This is the core secret behind SGLang being "unreasonably fast" on real workloads.

🔌 Analogy
Picture a team's shared, continuously-extended "filing tree" — a bit like editor autocomplete. Many documents open the same way: "Chapter 1, Company Overview…". The first author writes that opening into the tree as a few stored pages; the next author whose document starts word-for-word the same needn't recopy a thing — walking down the tree, they find the path "Chapter 1, Company Overview" already written and reuse those pages, branching off only where their text actually diverges to add their own new pages. Whoever starts like someone else gets that part for free; only the divergence needs fresh ink. That shared, collaboratively-extended filing tree is the spirit of RadixAttention.
🌍 The big picture
In one line: RadixAttention stores every request's KV in a radix tree (compressed prefix trie) keyed by the token-id sequence; a new request matches the longest prefix down the tree, reuses the matched KV with zero recompute, and inserts its divergent suffix as a new branch. The cache key is the token sequence itself, so sharing is fully automatic — no user hints, hitting across requests and across chat turns. Leaves are evicted by LRU but protected by reference counting: KV in use by a running request is never evicted. Tree nodes point at Lesson 6's paged physical blocks — sharing a prefix just means multiple paths share the same physical pages.

Real workloads share prefixes

First, see the waste clearly. Production requests are rarely isolated; their openings are highly identical, from roughly four sources: system prompts (every request carries the same few-hundred-token preamble), few-shot examples (the same demos prepended over and over), multi-turn chat (turn N resends all of turns 1..N−1 verbatim), and agent scaffolds (the same tool specs, format rules, persona up front). These openings run hundreds-to-thousands of tokens; recomputing their KV from scratch per request and per turn burns prefill compute in huge blocks on duplicate content.

Tie this back to Lesson 6: because paging turned KV into fixed-size, page-table-indexed physical blocks, "two requests reuse a prefix" collapses to two tree paths pointing at the same physical pages — trivially cheap. And it saves more than memory: the prefill compute for that shared prefix is gone too. The higher the hit rate, the bigger the win — that's where RadixAttention's value comes from.

RadixAttention: store KV in a radix tree

The core structure is a radix tree (a.k.a. compressed prefix trie): the key is the token-id sequence, and each edge carries a run of consecutive tokens plus the corresponding KV blocks. A new request does just one verb — match down the tree:

1

match_prefix: walk down the tree

Use the request's token sequence as the key; from the root, compare edge by edge to find the longest cached prefix. The deeper the hit, the more KV is free.

2

Reuse the matched prefix KV

The matched prefix's KV is reused with zero recompute — the attention kernel follows the nodes to the physical block indices, as if it always belonged to this request.

3

Insert the divergent suffix as a new branch

Everything past the divergence point is unique to this request; insert it as a new branch and compute/allocate fresh KV blocks for it.

4

Split an edge at the divergence if needed

If the match ends in the middle of an edge, split that edge in two: the front becomes a shared parent, the back and the new suffix each form a branch.

Splitting an edge is the essence of a radix tree. Suppose the tree already holds "You are a helpful assistant. Translate", and a new request only shares up to "You are a helpful assistant. " before diverging. No need to rebuild — just cut that long edge in two at the split: the common part becomes a shared ancestor; the leftover and the new suffix each hang off it. One cheap pointer operation lets two requests share down to the exact token boundary.

Automatic reuse across requests and turns

Because the cache key is the token sequence itself, sharing is fully automatic: hits happen not only across different requests but — even more so — between adjacent turns of one chat. Multi-turn works like this: turn N resends all prior turns verbatim, then appends the new question. So all KV of turns 1..N−1 is an already-cached prefix; match_prefix hits all the way to the end of history, and the engine only pays compute for the latest turn. The longer the chat, the longer the reused prefix, the bigger the savings.

This "works automatically, no cooperation required" property lets RadixAttention profit on any real traffic — the reason Lessons 1 and 3 keep naming it a signature SGLang feature. Paired with cache-aware scheduling (Lesson 20), the engine even actively groups same-opening requests together to push hit rates higher.

Why a tree, not a hash map — and how eviction works

You might ask: to check "do the openings match", couldn't a hash map from "whole prompt → KV" do it? Far from enough. Prefixes are hierarchical: requests A, B, C may share 100 tokens, of which A and B share 50 more. A hash map only knows whole-string equality — no longest-prefix match, no partial reuse, and no notion of "shared ancestor" or "cheap split". A radix tree is built for prefixes:

CapabilityHash map (whole-string)Radix tree (compressed trie)
Matchingonly hits an exact whole stringwalks down for longest-prefix match
Partial reusesame start, different end ⇒ full missreuse as deep as it hits, token by token
Hierarchical sharingcan't express a "shared ancestor"common prefix is a shared parent naturally
Divergenceonly adds a separate entrycheaply splits an edge at the fork

The cache fills up, so it needs eviction. RadixAttention evicts leaves by LRU: the least-recently-accessed leaf goes first, freeing physical blocks. But one iron rule applies — reference counting: each running request locks (lock_ref) the path it is using, and any node with refcount > 0 is never evicted. The reason is plain: a request is decoding against some prefix's KV right now; evict that KV and the request crashes on the spot. So eviction only touches leaves no running request references — safety and high hit rate at once.

def match_prefix(self, params: MatchPrefixParams) -> MatchResult:
    # Find the longest cached prefix of `key` in the radix tree.
    key = params.key
    key, _ = key.maybe_to_bigram_view(self.is_eagle)

    if self.disable or len(key) == 0:
        return self._empty_match_result

    key = key.page_aligned(self.page_size)        # truncate to a multiple of page_size

    # walk down the tree; get matched-prefix KV indices + terminal node
    value, last_node = self._match_prefix_helper(self.root_node, key)
    if value:
        value = torch.cat(value)                  # concat matched KV index segments
    return MatchResult(
        device_indices=value,                     # reusable prefix KV indices (may be empty)
        last_device_node=last_node, last_host_node=last_node,
        best_match_node=last_node,
    )

Read this and you've got RadixAttention's soul: one match_prefix computes exactly how much KV this request gets for free. The returned device_indices are the physical block indices of the reusable prefix — possibly length 0 (a brand-new request), possibly hitting all the way to the end of history (turn N of a chat). The docstring's note that a match ending inside a segment splits a node once is exactly the cheap split above. Stack this with Lesson 6's paging and Lesson 5's continuous batching and you see SGLang's three-layer memory foundation: per-slot (L4) → paged (L6) → prefix-shared (this lesson).

Back to the main thread: Lesson 4 built the KV pool, Lesson 5 kept the batch full, Lesson 6 packed it tight with paging — this lesson lets different requests and turns share the same prefix KV, killing both duplicate compute and duplicate storage. Later, cache-aware scheduling (Lesson 20) actively matches up same-opening requests to raise hit rates, and HiCache, which tiers the prefix cache across HBM/host/disk, comes in Lessons 29–32. Read on with this yardstick — the token sequence is the cache key; the longest prefix is reuse — and you'll find SGLang's real-traffic throughput nearly all springs from this humble radix tree.

📌 Key points
  • Pain: real workloads heavily share prefixes — system prompts, few-shot examples, multi-turn chat (each turn resends all history), agent scaffolds. Recomputing that opening's KV per request/turn is huge waste.
  • RadixAttention: store all KV in a radix tree (compressed prefix trie) keyed by token ids; match_prefix reuses the matched prefix KV with zero recompute, inserts the divergent suffix as a new branch, and splits an edge when the match ends mid-edge.
  • Automatic: the cache key is the token sequence, so hits happen across requests and across chat turns with no user hints; tree nodes point at Lesson 6's paged physical blocks, sharing = many paths reusing the same pages.
  • Eviction: leaves go by LRU but are guarded by reference counting — KV in use by a running request is never evicted, or that request would crash.
  • Why a tree not a hash map: prefixes are hierarchical; a tree gives longest-prefix match, partial reuse, shared ancestors, cheap splits; a whole-string hash only knows exact equality and misses entirely when starts match but ends differ. Throughput vs latency

From Lesson 4 to 7 we stacked tricks for "faster and cheaper": continuous batching keeps the GPU fed, paging packs HBM tightly, RadixAttention shares prefixes. But every one of these must answer one unavoidable question: do you want to serve the most requests per second, or return each request the fastest? You cannot max both. This lesson closes Part 2 and is the runtime's compass — only once you grasp the throughput vs latency balance does the scheduler (Lesson 18) make full sense.

🔌 Analogy
Picture a restaurant torn between two philosophies. First, "make every table fast": seat a guest, serve instantly, they leave — per-table wait is tiny, but the kitchen only handles a table or two at a time, so total people fed per hour stays low. Second, "feed the most people": gather a big group, start a seating together, cook one big batch to amortize cost — hourly capacity is maxed, but any single diner queues longer and waits for a bigger batch to finish. "Serve fast" is low latency, "feed many" is high throughput, and the "how many per seating" knob sacrifices one end whichever way you turn it.
🌍 The big picture
In one line: throughput and latency are two ends of one curve, and batch size is the slider. Bigger batches amortize the weight read and saturate the GPU ⇒ higher total throughput; but each request waits behind more work and each step is heavier ⇒ higher per-request latency. Smaller batches flip it: low latency, low throughput, idle GPU. No free lunch — you can only pick a point on the curve for your SLA. SGLang's scheduler, memory pool, and chunked prefill are all tools to help you pick that point well.

Before trading off, you need measurable rulers. Serving has four core metrics, each dominated by a different bottleneck; telling them apart is the prerequisite for understanding any tuning:

Their enemies differ — TTFT fears a long queue or one giant prompt blocking everyone; ITL fears a batch so big that each step drags. A system that crushes TTFT (small batch, serve now) may have poor ITL and throughput; one with sky-high throughput (gather, then send) may have ugly TTFT. What you truly maximize is throughput under the SLA.

Why do big batches raise throughput? Back to Lessons 4–5: decode is memory-bound, and the cost of reading weights once is nearly the same for 1 request or 100; a bigger batch amortizes that fixed cost thinner, yielding more useful tokens per unit of compute and pushing the GPU toward full load. Why do big batches hurt latency? Because one decode step now computes for more requests, so the step takes longer (ITL rises), and a new request waits behind more work before its turn (TTFT lengthens). That's the balance — the same move that feeds throughput slows each request. So there is no universal "optimal batch size", only the optimal operating point for your SLA.

Different loads, different balancing tools

Real traffic isn't one shape. Prefill-heavy (long prompt, short output, e.g. doc QA) and decode-heavy (short prompt, long output, e.g. long-form generation) pull the balance in different directions and need different tools:

Chunked prefill (Lesson 22)

One giant prompt's prefill can hog a whole step and stall everyone's decode — showing up as a sudden latency spike (ITL jitter). Chunked prefill slices the long prefill and interleaves it with decode steps, so neither throughput nor latency collapses over one long request.

PD disaggregation (Lesson 45)

Prefill and decode have different bottlenecks (compute- vs memory-bound). PD disaggregation splits the two stages onto separate GPU pools, each batched and tuned independently — prefill chases throughput, decode protects latency, without dragging each other.

The structured-generation language

The previous parts were all about the runtime — KV cache, continuous batching, paging, prefix reuse. But the project's name, SGLang = Structured Generation Language, really lands on the word Language: first and foremost it is a front-end language for writing "calling an LLM" as a program. This lesson returns to the project's namesake origin and its core shape — an ordinary Python function decorated with @sgl.function, where gen leaves "blanks to fill" and select leaves "multiple-choice blanks," turning a multi-step dialogue into a readable, controllable, reusable program.

🔌 Analogy
An SGLang program is like a fill-in-the-blank form / mail-merge template: you write the fixed text ahead of time and leave labeled blanks wherever the model should answer. A plain blank (gen) lets the model write freely; a multiple-choice blank (select) offers only a few preset options, so the model can only pick one. You design the form's skeleton and slots; the model fills the blanks — and afterward each blank carries a label, so s["label"] pulls the answer back out for reuse.
🌍 The big picture
In one line: SGLang lets you write "multi-call LLM" programs using ordinary Python control flow. The function takes a state object s; you build the prompt with s += sgl.user("…") and insert generation slots with gen(...), with arbitrary if / for / plain Python between two gens. This buys three things: ① structure and control flow live in Python — clear and debuggable; ② constrained decoding (select, regex, json) makes outputs reliable and parseable; ③ the runtime automatically reuses KV across shared prefixes (Lesson 7, RadixAttention).

Why a DSL instead of hand-rolling a raw API?

For the same "call the model twice, feed the first result into the second," hand-writing raw HTTP / OpenAI API means you must concatenate strings, parse, manage history, and enforce format yourself; SGLang sinks all that into the language and runtime, so you write only "what to do" and leave "how to do it efficiently and reliably" to the framework.

Raw API: what the framework auto-does

① KV across shared prefixes is reused automatically (Lesson 7), no manual caching; ② constrained decoding guarantees valid select/json output; ③ scheduling, continuous batching, paging all happen transparently in the runtime. With a raw API you'd do all of it yourself.

The third benefit is the subtlest and most valuable: when you write the program structurally, the framework can see the structure and optimize from it. If multiple requests share a prefix (same system prompt, same few-shot examples), the runtime computes the prefix once and reuses its KV; when one function forks into branches (Lesson 11), the branches share the parent's prefix cache. In other words, writing a structured program is itself feeding optimization hints to the runtime — something the raw-API style of "fire one, read one, blind to each other" can never obtain.

Interpreter & tracer

Last lesson we saw what an SGLang program looks like — an ordinary Python function decorated with @sgl.function, leaving blanks via gen / select, essentially a "tree of intent to be executed." This lesson answers: how does that tree actually run? SGLang has two ways to realize a program — interpret, where StreamExecutor really drives the backend step by step and generates text; and trace, where trace_program walks the function symbolically WITHOUT calling the model, just to see the structure and extract the static prefix. Grasp these two modes and you hold the main axis of the SGLang front end — from "written" to "run."

🔌 Analogy
Think of an SGLang program as a recipe. You can use it two ways. First, cook it step by step right now (interpret) — chop, sear, stir, and each step really produces food; how far you've gotten and what's in the pan are concrete results. Second, read it through first (trace) — you don't turn on the stove; you just follow the fixed leading steps to see what's needed, so you can prepare a shopping list and mise en place ahead of time. You stop at the first step like "taste it, then decide how much salt" that depends on a live result — that one you can only know by actually cooking. Interpret = do it now; trace = understand the shape first and pre-stage what you can.
🌍 The big picture
In one line: the same tree of intent can be "executed" or "analyzed." Interpret is the normal path — StreamExecutor takes the function and drives it expression by expression: fixed text is appended to the state; a gen / select calls the backend (Lesson 12), streaming the model's output onto the text and into a named variable; the if / for between two gens is plain Python that runs normally. Trace is the compile/analysis path — trace_program walks the function symbolically with a dummy backend, never touching the model, just to learn the program's structure and extract the constant leading prompt (the static prefix), which the runtime can precompute, cache, and share (echoing Lesson 7, RadixAttention). One answers "what to produce now," the other answers "what shape is this program" — that is the division of labor between interpreter and tracer.

The interpret path: how StreamExecutor drives a program

When you call an @sgl.function, SGLang builds a StreamExecutor and a ProgramState (the s in your function). The executor runs a worker in a background thread: every s += … in your body essentially drops an expression (expr) into its queue, and the worker _executes them one by one. Fixed text is appended to text_; a gen calls backend.generate(...) and appends the result to text_ and stores it in variables[name]; a select asks the backend to pick one of the choices. The key idea: interpreting is "produce as you go" — text and variables really grow step by step.

Trace is the other road. trace_program / extract_prefix_by_tracing builds a TracerProgramState and a dummy backend (BaseBackend), then walks your function symbolically — it never calls the model. The heart of the interpreter is StreamExecutor. In a background thread it holds the backend, the variables table, the full text text_, and more; submit drops an expression into the queue, the worker pops it and _executes, and _execute_gen is where the model is actually called and the result is written back into text_ and variables. class StreamExecutor:
    """A stream executor that executes SGL expressions in a background thread."""

    def __init__(self, backend, arguments, ...):
        self.backend = backend          # the chosen backend (Lesson 12)
        self.variables = {}             # named results: s["name"] reads from here
        self.text_ = ""             # the ever-growing full text = the prefix
        self.messages_ = []             # role messages in OpenAI format
        # …a background worker thread consumes expressions from the queue…

    def submit(self, expr):             # drop a gen/select node into the queue
        self._init_var_event(expr)
        self.queue.put(expr)

    def _execute_gen(self, expr):       # where the model is actually called
        comp, meta_info = self.backend.generate(self, ...)
        self.text_ += comp              # append the generated result to the text
        self.variables[expr.name] = comp  # store into a named variable for s["name"]

Read this and you grasp the interpreter's soul: it is a loop that "consumes expressions one by one and keeps accumulating results into state." Fixed text is appended; gen/select hits the backend and writes back into text_ and variables — that's how state grows step by step. The tracer walks the same tree but swaps "hit the backend" for a symbolic placeholder, so it can see the structure and extract the prefix without spending a single FLOP. fork/join & prefix sharing

Last lesson (Lesson 10) showed how the interpreter runs a tree of intent step by step. But what really sets SGLang apart is that a program can fork multiple parallel branches off the current state — that is s.fork(n). The one line to remember today: the front-end's fork and the runtime's RadixAttention (Lesson 7) are two halves of one idea. fork lets you declare "branch here" in the program; RadixAttention computes that branch's shared prefix once in the engine and reuses it for every branch. One states the structure, the other cashes it in as saved compute. Grasp this correspondence and you finally understand the weight of the word Structured in SGLang's name.

🔌 Analogy
fork/join is like a choose-your-own-adventure book: every reader first reads the same opening chapters (the shared prefix); at a fork in the road the book says "to save the princess turn to page 50, to slay the dragon turn to page 80" — so one story branches into several different endings. The key: the shared opening is printed only once, never re-copied for each branch; each reader only reads their own short second half. Finally you lay the endings side by side and pick the best — that is join (gathering each branch's result back). fork = branch at the shared opening, join = aggregate the separately-obtained answers.
🌍 The big picture
In one line: s.fork(n) clones n parallel continuation branches from the current state, each inheriting the same prompt prefix built so far, then each generates independently (e.g. n different samples, or n different sub-questions); join / gather then collects each branch's newly-captured variables back into the parent state for you to compare, vote, or aggregate. Its power is not convenience but saved compute: since the n branches share an identical prefix, RadixAttention computes that prefix's KV once and shares it across all branches (Lesson 7). So forking 8 branches off a long prefix costs roughly "1 prefix + 8 very short suffixes," not "8 full prompts." fork is therefore not just a convenience — it is what "prefix sharing" looks like in the programming model.

What fork does: N branches grown from one state

Think of state s as an ever-growing dialogue record (Lessons 9, 10). At some moment you have built up a long prompt into s — system setup, few-shot examples, the user question. Calling s.fork(n) now clones n identical copies at the current tail of that record, each carrying the entire prefix so far. The n branches then go their separate ways: branch 0 may sample at one temperature, branch 1 at another; or you append a different sub-question to each. They are independent and do not interfere. Once all branches finish, you call join (default gather_variable mode), and the framework copies each branch's newly-written named variables back into the parent state s — e.g. a variable in the parent becomes a list holding all n branches' answers, ready to compare, vote, or pick the best.

The core: fork and RadixAttention are two halves of one idea

This is the part to highlight hardest. Lesson 7 explained: RadixAttention organizes all requests' prefixes into a radix tree, where a common prefix stores one copy of KV shared by many requests. What fork does is exactly deliberately create a "one common prefix + many different suffixes" shape in the program. In other words: fork is the "declaration side," RadixAttention is the "redemption side." Writing s.fork(8) in the DSL tells the runtime plainly: "these 8 branches have an identical opening." The runtime takes that signal and computes the shared prefix's attention only once; the 8 branches share one prefix KV, each adding compute only for its own short suffix. Without fork's declaration, RadixAttention could still discover after the fact that prefixes match; with fork, the sharing is structurally explicit, zero-guess.

Serial loop vs fork: where the cost differs

For "best-of-n," the naïve way is a for loop running n times: each iteration resends the full prompt. The problem: those n requests share an identical long prefix, yet it gets recomputed n times end to end — burning (n−1) prefixes of compute, and the iterations run serially, blind to one another. With s.fork(n), you declare the "shared prefix + n suffixes" structure once, the runtime computes one prefix plus n short suffixes, and independent branches can run concurrently. The longer the prefix and the larger n, the more you save.

Serial for loop: N × full prompt

Run n gens in a loop, each recomputing the whole long prefix. Cost ≈ n × (prefix + suffix), prefix recomputed n times; iterations serial, runtime can't tell their openings match.

s.fork(n): 1 × prefix + N × suffix

Declare the fork once; the runtime computes the shared prefix KV only once (Lesson 7), n branches each add only a short suffix. Cost ≈ prefix + n × suffix; independent branches also run concurrently (Lesson 10).

python/sglang/lang/interpreter.py ::ProgramState.forkfork: branch N sub-states off the current state
def fork(
    self,
    size: int = 1,
    position_ids_offset: Optional[List[int]] = None,
):
    # Have the executor clone `size` branches from the current state — they share the built prefix
    stream_executors = self.stream_executor.fork(size, position_ids_offset)
    states = [ProgramState(x) for x in stream_executors]
    # Pack into a group; later group.join() copies each branch's variables back to the parent
    state_group = ProgramStateGroup(states, self)
    return state_group

A few lines, but they tie the two halves together: stream_executor.fork(size, …) does the real "clone size branches from the current prefix" at the runtime layer — exactly the moment RadixAttention's prefix tree grows size children off a shared parent node; the returned ProgramStateGroup gives you join, which in gather_variable mode copies each branch's new variables back into the parent state. With one s.fork(n), front-end declaration and runtime reuse become one and the same. Lesson 12 next covers the backend interface, landing this fork/join onto concrete engines — completing Part 3's chain of "language → interpret → fork → land."

📌 Key points
  • What fork does: s.fork(n) branches n parallel continuations off the current state, each inheriting the same already-built prefix, then generating independently; join (gather) copies each branch's new named variables back into the parent for compare / vote / aggregate.
  • The core correspondence: fork is the "declaration side," RadixAttention (Lesson 7) the "redemption side" — two halves of one idea. fork states "these branches share an opening"; the runtime computes the shared prefix's KV once and reuses it for every branch.
  • Cost: forking 8 branches off a long prefix ≈ 1 prefix + 8 short suffixes, not 8 full prompts; a serial for loop recomputes the prefix n times. The longer the prefix and larger n, the more you save.
  • Common uses: best-of-n sampling, branch-and-evaluate, map over a list, tree-of-thought — all share "share an opening, then do things separately."
  • Concurrency: branches with no data dependency can be advanced concurrently by the interpreter (Lesson 10). So fork saves compute (prefix reuse) and time (concurrency) at once. Source: interpreter.py ::ProgramState.fork; Lesson 12 lands it on concrete backends.

Backends & OpenAI compat

From Lesson 9 through Lesson 11 we have been writing programs: describing intent in the DSL (Lesson 9), letting the interpreter run the tree of intent (Lesson 10), forking parallel branches with fork/join (Lesson 11). But one question has been quietly dodged all along — who actually executes all these gen, select, fork calls? The answer is today's star: the Backend. The heart of SGLang's design is that the same front-end program runs unchanged on different backends — the local SGLang runtime, OpenAI, Anthropic, VertexAI… and the program itself has no idea which backend it runs on. This lesson closes Part 3 (the front-end) and, through the RuntimeEndpoint local-runtime backend, snaps seamlessly onto the runtime world of Part 4+: your fork/gen become HTTP calls to the scheduler we study next.

🔌 Analogy
A backend interface is like a universal remote: one set of buttons (power, channel, volume) drives any TV — the living-room one, the bedroom one, a friend's. When you press "channel," you needn't care what brand sits in front of you. But there is a key difference: only your own TV (the local runtime) exposes the advanced features — picture enhancement, recording, scheduling (i.e. prefix cache, constrained decoding, branch sharing); a generic stranger's TV (a hosted API) only knows the basic buttons. "One button set drives all TVs" maps to the BaseBackend abstract interface; "only your own TV has the advanced keys" maps to: only the local runtime gives you RadixAttention and constrained decoding, while hosted models are black boxes — whether they cache, whether they strictly constrain, is not up to you. Remember this remote and you remember the whole lesson: one interface, layered capabilities.
🌍 The big picture
In one line: BaseBackend is an abstract interface that prescribes what a backend must be able to do — generate, select (pick among candidates), streaming, role handling, and so on. Your DSL program calls only these abstract methods and never touches a concrete engine directly. So switching backends is one line, set_default_backend(...): pass RuntimeEndpoint("http://localhost:30000") to run on the local SGLang runtime (the /generate HTTP server we build in Lessons 13–17), pass OpenAI(...) and the same program targets OpenAI's hosted models instead. But there is no free lunch: the local runtime gives you RadixAttention prefix caching (Lesson 7), constrained decoding, fast batching; a hosted backend buys you portability at the cost of losing prefix-cache and fork-sharing benefits — those models are black boxes, caching is the provider's call, and constrained decoding is limited.

1. BaseBackend: the seam that decouples program from engine

Recall Lesson 10: the interpreter (StreamExecutor) executes the expressions of the intent tree one by one — fixed text is appended, and a gen / select issues a call to the backend. That "backend" is any class inheriting from BaseBackend. BaseBackend abstracts "an engine that can execute an SGLang program" into a set of methods: how to generate tokens, how to select among candidates, how to stream, how to handle system/user/assistant roles. The program always faces this interface, never a specific engine. This is the classic "program to an interface": the upper layer depends on a contract, not on the lower layer's implementation. So the same intent tree can run on local GPUs today and a cloud API tomorrow, with not one character changed.

3. Local runtime vs hosted API: what are you trading?

Switching backends is not free. RuntimeEndpoint and OpenAI look identical in the program, but the capabilities you get differ vastly. The crux: the local runtime is a white box you can see through and control, so the prefix cache (Lesson 7), fork sharing (Lesson 11), and strict constrained decoding (guaranteeing valid JSON / regex output) are all available; while a hosted model is a black box — the provider may cache internally, but you cannot declare or rely on it, and constrained decoding usually degrades to "begging for it in the prompt." Here is the trade laid out.

Local runtime (RuntimeEndpoint)

White box, controllable: full RadixAttention prefix cache (Lesson 7), fork branch sharing (Lesson 11), strict constrained decoding, fast batching. The cost: you must deploy and manage GPUs yourself (Lessons 13–17 teach you). For performance and control, use this.

Hosted API (OpenAI / Anthropic …)

Black box, portable: one line switches to a top closed model, no GPUs needed. The cost: you lose the deterministic prefix-cache and fork-sharing benefits, constrained decoding is limited, and you pay per token. For convenience and portability, use this.

class RuntimeEndpoint(BaseBackend):
    def __init__(self, base_url, api_key=None, ...):
        super().__init__()
        self.base_url = base_url
        # Startup handshake: ask the local runtime which model is loaded
        res = http_request(self.base_url + "/get_model_info", ...)
        self.model_info = res.json()

    def cache_prefix(self, prefix_str):
        # Send the shared prefix with max_new_tokens=0 to pre-warm RadixAttention (Lesson 7)
        http_request(self.base_url + "/generate",
            json={"text": prefix_str, "sampling_params": {"max_new_tokens": 0}})

    def generate(self, s, sampling_params):
        # A gen in the program becomes one POST /generate HTTP call here
        data = {"text": s.text_, "sampling_params": {...}}
        res = http_request(self.base_url + "/generate", json=data)
        return res.json()["text"], res.json()["meta_info"]  # completion text + meta

These few lines are the door from front-end to runtime: the constructor handshakes via /get_model_info to confirm the model; cache_prefix uses max_new_tokens=0 to pre-warm the shared prefix into the cache, the very mechanism behind fork's compute savings (Lesson 11); and generate faithfully translates each gen in the program into one POST /generate. Understand this and you understand the seam between Part 3 and Part 4 — from the next lesson on, we step into the server behind that /generate, watching TokenizerManager, Scheduler, and ModelRunner turn this one HTTP request into real tokens on the GPU.

Engine & HTTP server

Welcome to Part 4 — we finally step from the front-end DSL through the door of the runtime. This lesson answers the most basic question: through what entry point does your program feed requests into SGLang's engine? There are exactly two answers: the offline Engine (a pure in-process Python API) and the online HTTP server (a FastAPI app). Understanding these two doors — and that they share one runtime underneath — is the key to every later lesson.

🔌 Analogy
Picture a restaurant with two ways to order. One is the walk-in counter: you step to the window and call your order straight to the kitchen — no phone, no middleman. That's the offline Engine: an in-process call with the lowest overhead. The other is the phone-order hotline: you dial, give your address, and an operator relays the ticket to the kitchen. That's the online server: a layer of network and protocol, but anyone, any client, can order remotely. The point: there is only one kitchen. Counter or phone, the kitchen is identical — the server is just a "front desk that answers calls" bolted onto the same Engine.
🌍 Big picture
SGLang offers two paths to drive the runtime, both inheriting from the same abstract base EngineBase: the offline Engine is a plain Python object whose llm.generate(...) returns results directly — ideal for batch jobs, evaluation, and RL rollout (the trainer calls it in-process); the online server uses launch_server() to start a FastAPI/uvicorn app wrapping an Engine, exposing the native POST /generate plus OpenAI-compatible routes. The runtime underneath is identical: the server is just an HTTP wrapper around the Engine.

Offline Engine: a pure in-process Python API

The offline Engine is the barest entry: no HTTP, no port, no network. In your own Python script you import sglang as sgl, build a sgl.Engine(model_path=...), then call llm.generate(prompts, sampling_params) — results come back right away as Python dicts/lists. It all runs in one process (precisely: the main process also holds the TokenizerManager, with the scheduler and detokenizer as subprocesses — more on that soon). With no serialization, HTTP parsing, or network round-trip, the Engine is the lowest-latency, most embeddable way in.

Offline Engine

sgl.Engine(model_path=...) + .generate(...). Pure in-process call, no HTTP, no network round-trip. Best for batch inference, offline eval, RL rollout (the trainer calls it in-process, lowest overhead). Downside: usable only from Python in this process — no cross-machine/cross-language.

Online server

launch_server(server_args) starts a FastAPI app exposing an HTTP interface. Best for production serving, multi-client, language-agnostic access — any HTTP client connects. Cost: an extra network + serialization layer, plus ports, concurrency, and auth to manage.

Online server: wrap the Engine in HTTP

The online server starts with one command: python -m sglang.launch_server --model-path .... Internally it builds an Engine, then wraps it with FastAPI/uvicorn, exposing two kinds of routes: SGLang's native POST /generate, and a full set of OpenAI-compatible routes like /v1/chat/completions and /v1/completions (Lesson 15 covers the compat layer). TokenizerManager

Last lesson said: whether you enter through the offline Engine or the online server, requests land on one shared runtime. So once a request is inside, which component catches it first? That is today's star — the TokenizerManager. It is the runtime's front door: it runs in the main process (alongside the HTTP server / Engine), not in the GPU-hungry subprocess. Every request is first tokenized, packed and registered here before being sent on to the scheduler.

Why give it its own lesson? Because the TokenizerManager is the translator between the outside text world and the inner token world, and the async concurrency hub. You send a string; the engine speaks token ids. Hundreds of requests pour in at once — whose reply goes back to whom is its job to coordinate. Understanding this front door is what lets Lesson 16's io_struct messages, Lesson 17's detokenize/streaming, and Lesson 18's scheduler chain into one coherent flow.

Think of the TokenizerManager as a hotel front-desk receptionist. A guest (request) walks up and states their need in spoken words (raw text). The receptionist translates it into a standardized ticket — fills in the room type and requests (sampling params), staples on a booking number (rid), and sends the ticket to the back office (the scheduler subprocess) to actually arrange things. The guest never enters the kitchen; they just wait to be called. Meanwhile the receptionist juggles dozens of guests at once, handing each reply back to exactly the right person. The desk never cooks (never touches the GPU): it only translates, registers, forwards, awaits, and delivers — exactly the TokenizerManager's job.
🌍 Big picture
The TokenizerManager lives in the main process as the CPU-side async coordinator. Its core method generate_request is an async coroutine: it tokenizes the text into token ids (pure CPU string work), builds the sampling params, packs everything into a TokenizedGenerateReqInput (Lesson 16) carrying a unique rid, and sends it over ZMQ to the scheduler subprocess. It then registers a per-request async state keyed by rid and awaits the outputs streaming back from the detokenizer (Lesson 17), yielding them to the caller (HTTP SSE or the Engine). Isolating tokenize/detokenize from the GPU loop in separate processes is what lets CPU and GPU overlap (the zero-overhead idea, Lesson 21).

Which process? Why is it CPU work

Recall Lesson 2's three-process model: tokenize in the main process, schedule+forward in the scheduler subprocess, detokenize in the detokenizer subprocess. The TokenizerManager sits in the main process slot — same process as the HTTP server / Engine, not in the GPU subprocess. This is a deliberate process-boundary design, not an accident.

The key: tokenize and detokenize are CPU-bound Python string work — table lookups, BPE merges, special-token handling, all CPU, nothing to do with GPU matmuls. If you crammed them into the GPU subprocess, the GPU would idle while the CPU tokenizes and vice versa — the two would serialize and drag each other down. By locking the GPU loop in its own process and letting the TokenizerManager do the CPU-side preprocessing in the main process, the two can truly overlap: while the GPU forwards the previous batch, the CPU is already tokenizing the next. This is the foundation of SGLang's zero-overhead scheduling (Lesson 21), and the TokenizerManager ↔ scheduler hop is exactly the ZMQ/IPC seam.

The rid round-trip: how the hub routes replies

The TokenizerManager's cleverest part is its async routing. Hundreds of requests arrive concurrently, each assigned a unique rid; after sending, the main process does not block — it records rid → coroutine state in a map (rid_to_state). Without the rid, hundreds of concurrent replies would go to the wrong callers. It also handles abort and, for multimodal, image/audio preprocessing.

OpenAI/Anthropic/Ollama compat

Last lesson we saw the runtime's front door — the TokenizerManager. But one question it can't answer: out there are clients speaking the OpenAI dialect, the Anthropic dialect, the Ollama dialect — how can they plug into one runtime without changing a line? Why a whole lesson on the compat layer? Because it is the seam that lets SGLang drop into the existing ecosystem. You don't rewrite an SDK for SGLang, nor make your team learn a new API — just point an OpenAI client's base_url at your SGLang deployment and it runs immediately. LangChain, the openai Python package, any tool built around the OpenAI API — all plug in seamlessly. This thin adapter layer is the master key by which a private deployment takes over the whole ecosystem.

🔌 Analogy

A request's protocol journey

Walk the whole path in your head first. A request's journey is: client SDK → protocol adapter → native request → the same runtime → adapter formats the response. Note the middle stretch — "the same runtime" — whichever dialect the entry spoke, it converges here onto one shared fast path. The adapter only translates at the outermost layer; it never reimplements the engine.

That is the essence of the compat-layer design: the runtime itself is protocol-agnostic — it only knows GenerateReqInput. The adapter does just two things — schema translation (map OpenAI/Anthropic/Ollama fields onto native fields) and chat templating (assemble the messages list into a single prompt per the model's template). Because its job is so narrow, adding a protocol is cheap: write one new adapter. The hottest inference fast path stays shared and optimized once. If every new protocol meant rewriting the engine, that would be an engineering disaster — the compat layer sidesteps it elegantly with "thin adapters + shared core."

IO structs & IPC

Lesson 14 showed the front door (TokenizerManager) and Lesson 15 the multi-protocol translators, but one thing kept slipping by: these three processes do not share memory — so what exactly travels between them? That is today's star: the group of message structs in io_struct.py. Processes cannot read each other's variables; they can only pass messages. And io_struct.py pins down the shape of those messages as Python dataclasses, serialized over ZMQ.

Why a whole lesson for it? Because this set of structs is effectively the runtime's "type system" and wire protocol. Every later component — the scheduler (Lesson 18), the worker, the detokenizer (Lesson 17) — talks to the others in these structs. Grab the single thread of "how a request changes shape across processes," and those seemingly complex components string together like beads. Skip it, and the scheduler source will drown you in TokenizedGenerateReqInput and BatchTokenIDOutput — they are not details, they are the skeleton.

There is a deeper angle too: making communication explicit as data structures is itself an engineering discipline. Once the process boundary is pinned by dataclasses, "who sends which field, who fills it, who reads it" becomes black and white — printable, breakpoint-able, unit-testable. Compared with implicit shared state, this "everything is a message" design costs a few extra classes but buys debuggability and evolvability — a lesson distributed systems keep re-teaching.

🔌 Analogy
Think of these structs as standardized forms and envelopes moving between departments. For one case, each department turns it into a new official form: the front desk takes an intake form (raw ask: text, sampling prefs, stream flag); tokenization turns it into a work order (token ids + params, sent to the floor); the floor produces a results sheet (a batch of token ids); finally it is typed up as a printed letter (decoded text) and mailed back. Every form is stamped with the same case number (rid) in the corner, so even with dozens of cases interleaving on the conveyor belt, nothing is misrouted. Departments never share a drawer (no shared memory) — they cooperate purely by handing forms across (passing messages).
🌍 Big picture
io_struct.py lives under python/sglang/srt/managers/ and is a pure message-definition file: almost no algorithms, just @dataclass after @dataclass. They line up along the request's lifecycle — GenerateReqInput (the user-facing request the front door receives) → TokenizedGenerateReqInput (post-tokenization, the one that actually crosses ZMQ to the scheduler) → BatchTokenIDOutput (batched token ids from scheduler/worker) → BatchStrOutput (text decoded by the detokenizer). Each struct marks who produces and who consumes it. They travel over ZMQ IPC sockets (a port per process), and the rid routes interleaved batched outputs back to the right request. There are also standalone control messages: abort, flush_cache, health.

Why message passing, not shared memory

Recall the three-process model (Lesson 2): the main process (TokenizerManager), the scheduler subprocess, the detokenizer subprocess. They are three separate OS processes, each with its own memory — an object in the main process is invisible and untouchable to the scheduler. This is deliberate: splitting processes sidesteps Python's GIL and lets CPU and GPU truly overlap (Lesson 21). The price: they can no longer cooperate by "reading the same variable" — only by handing messages back and forth.

So the question becomes: what do messages look like, and how do they cross the wire? SGLang's answer is plain and effective — define shape with dataclasses, do transport with ZMQ. Dataclasses make every field obvious and printable; ZMQ provides lightweight inter-process sockets that serialize an object to bytes, send it, and deserialize it on the other side. Together they yield communication that is cheap, explicit, and debuggable — you can literally print a message and see its tokens and sampling params.

Shared memory? No

The three processes each own separate memory; one process's objects are unreadable to another. That is the boundary cost of splitting processes to dodge the GIL and overlap CPU/GPU (Lesson 21).

Message passing? Yes

Cooperation is purely by passing messages: shape with dataclass, cross the wire via ZMQ serialization. Cheap, explicit, printable and breakpoint-able — the classic distributed choice.

Without rid, the TokenizerManager could not split and route that interleaved batch back to the right awaiting coroutines.

  • It is the runtime's "type system": scheduler, worker, and detokenizer all speak these structs; abort/flush_cache/health are standalone control messages.

Detokenizer & streaming

Lesson 14 covered the runtime's entrance — the TokenizerManager turns human text into token ids and sends them to the GPU. This lesson covers the exit: how the token ids the GPU emits one at a time become readable text again, then stream back to the client piece by piece, like a typewriter. The component responsible is its own subprocess — the DetokenizerManager. It is the last component in the chain that speaks human language, and it closes Part 4.

Why give it its own process and its own lesson? Because stitching token ids back into text is far more than a table lookup. It solves two genuinely tricky problems: first, incrementality — each step must emit only the small new slice, not re-decode and re-send the whole generated text every time; second, multi-byte boundaries — a token may decode to only half a Chinese character or half an emoji, so it must wait for a complete character before emitting, never feeding a half-character to the screen. Grasp these two and you'll see why streaming looks so smooth while quietly relying on careful offset bookkeeping.

🔌 Analogy
Think of the detokenizer as a simultaneous interpreter. The speaker (GPU) keeps spilling out words; every few seconds the interpreter opens their mouth — but speaks only the words that appeared since they last spoke, never replaying the whole speech (that's sent_offset remembering "how far I've spoken"). Better still: when a syllable is only half out, the interpreter holds back and waits for the full word before voicing it — never a half-syllable (that's the "hold incomplete tokens" handling). So the audience always hears coherent, complete, never-repeated sentences. That's exactly the detokenizer's job: emit only the new, wait until whole, deliver by id.
🌍 Big picture
Recall Lesson 2's three-process model: tokenize in the main process, schedule+forward in the scheduler subprocess, detokenize in the detokenizer subprocess. After each step, the scheduler packs the batch's newly generated token ids into a BatchTokenIDOutput (Lesson 16) and sends it to the detokenizer; the detokenizer decodes ids into text, packs a BatchStrOutput, and sends it back to the TokenizerManager (Lesson 14), which routes it by rid to the awaiting coroutine, finally streaming it to the HTTP client via SSE. Detokenize gets its own process because it is pure CPU string work that can overlap with the GPU forward loop without either waiting on the other (the zero-overhead idea, Lesson 21).

Where it lives: the runtime's last door

The detokenizer is the pipeline's exit gate. It only speaks token ids and never touches the GPU: input is the batches of new tokens from the scheduler, output is slices of readable text. It mirrors the tokenizer exactly — one splits text into numbers (in), the other stitches numbers back into text (out). Note it sends results back to the TokenizerManager, not the client: only the TokenizerManager's rid_to_state map knows "which awaiting coroutine this rid belongs to and how to yield it to the upstream SSE".

Trace the exit path: the scheduler (subproc) finishes its forward and emits this step's new token ids; the DetokenizerManager (subproc) decodes ids into text and does the incremental slice; the slice returns to the main process's TokenizerManager, routed by rid to the awaiting coroutine; and finally it streams piece by piece via SSE to the HTTP client, surfacing as a "typewriter". Four stages, three process boundaries, all stitched into a loop by the rid.

Incremental detokenize: the magic of sent_offset

This is the lesson's most central — and most overlooked — piece. The naive approach: every step, decode all tokens so far into the full text and send the whole thing. It looks simple but has two fatal flaws: first, resending the whole string each step makes traffic grow quadratically (O(n²)) with length; second, the client repeatedly receives duplicate prefixes and must itself figure out "which part is new".

The right approach is incremental: each request keeps an offset sent_offset recording "which character I've sent up to". After decoding the current full text output_str each step, emit only the new slice output_str[sent_offset:], then advance sent_offset to len(output_str). Next step, again emit only the new tail. Every character crosses the wire exactly once; traffic is linear.

1

receive new token ids

The scheduler delivers this step's newly generated token ids (in a BatchTokenIDOutput, Lesson 16), appended to the request's existing id sequence.

2

decode the full text

Decode the current id sequence into the full-text-so-far output_str (with multi-byte boundary handling).

3

slice only the new part

Take incremental = output_str[sent_offset:] — only the small slice that appeared since the last send.

4

advance the offset

sent_offset = len(output_str), recording "sent up to here" so the next step never re-sends already-sent characters.

5

pack as an SSE chunk

The new slice rides a BatchStrOutput back to the TokenizerManager, then streams to the client as one SSE chunk — the "typewriter" effect.

To make the difference between "resend everything" and "send only the new" obvious, compare them side by side:

❌ Re-decode all each step

Decode all tokens and send the whole text every step. The client keeps receiving duplicate prefixes, traffic grows quadratically (O(n²)) with length — especially wasteful for long outputs — and the burden of "finding the new part" is dumped on the client.

✅ Incremental slice (sent_offset)

Each step sends only the small new slice output_str[sent_offset:] and advances the offset. Every character crosses the wire once, traffic is linear, the client just concatenates, and it maps naturally onto an SSE stream.

Multi-byte boundaries: why hold back

Incremental slicing hides another difficulty: one token is not one character. A Chinese character or an emoji often takes several bytes — sometimes spanning several tokens — to complete. If a step decodes a tail that is an incomplete UTF-8 sequence (half a character, a broken emoji) and sends it as-is, the client flashes a garbled box �. So the detokenizer must hold back bytes that don't yet form a complete character and wait until more tokens arrive to complete it. Stop conditions and SSE: when generation ends

Streaming always needs an end. A request's generation halts under one of three conditions, and the detokenizer/scheduler trims the output at the stop boundary accordingly (sampling and stop details in Lesson 28):

Stop conditionMeaningEffect on output
EOS tokenthe model generated the end-of-sequence tokennatural finish, this request ends streaming

When generation ends, the detokenizer materializes the request's full text, calls trim_matched_stop to cut off the stop string, emits that last tail (again only output_str[sent_offset:]), and then deletes the request's decode state. Turning each new slice into an SSE stream is the TokenizerManager's job: it uses StreamingResponse with media_type="text/event-stream", pushing each chunk as one Server-Sent Event to the browser — that's the typewriter experience of text "popping out one bit at a time". 📌 Key points
  • Exit / own subprocess: the DetokenizerManager is the runtime's exit, running in its own subprocess (Lesson 2's three-process model); it only speaks token ids and never touches the GPU.
  • Message round-trip: the scheduler sends BatchTokenIDOutput (new token ids); the detokenizer decodes and returns BatchStrOutput (Lesson 16) to the TokenizerManager (Lesson 14), routed back by rid.
  • Incremental is key: each request keeps a sent_offset; each step emits only output_str[sent_offset:] then advances the offset — avoiding the O(n²) and duplicate prefixes of full resends.
  • Multi-byte boundaries: incomplete UTF-8 (half a Chinese character / broken emoji) must be held back until a full character forms — never stream half a character.
  • Stop and SSE: generation ends on EOS / stop string / max_new_tokens (Lesson 28); the TokenizerManager wraps each slice into an SSE "typewriter" via StreamingResponse. A separate process so it can overlap the GPU (zero-overhead, Lesson 21).
